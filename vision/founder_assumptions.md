# Founder Assumptions

## Purpose

This file tracks the assumptions that can make or break Chief of Staff as Ty's trusted external memory system.

The riskiest questions are not only technical. They are about whether Ty will use this every day, trust it with real context, and return often enough for memory to compound.

If the system becomes useful for Ty for six months, company questions can be revisited from a stronger position.

## Primary Personal Assumption

This project is for Ty first.

The product should answer:

- Will Ty use this every day?
- Will it save Ty time?
- Will it remember things Ty forgets?
- Will it reduce Ty's mental load?
- Will it work across Ty's devices?
- Will Ty trust it enough to dump important information into it?

The startup question is secondary:

- What market?
- What TAM?
- What moat?
- What ICP?

Those questions matter later, after the system proves personal daily usefulness.

## Core Product Assumptions

- People want proactive suggestions.
- People trust AI with personal data.
- People want one inbox.
- People will dump information daily.
- Relationship graphs create value.
- Memory becomes more useful over time.
- Users prefer AI organization over manual organization.
- Users can understand and trust explainable memory.
- Users will correct the system when it is wrong.
- One suggested action can be more valuable than a complex dashboard.

## Primary Builder Risk

The biggest builder risk is feature creep.

The second-biggest risk is designing for investors, markets, or abstract users before proving personal utility.

Every new recommendation should be tested against one question:

Does this make the one-inbox memory system better?

If not, defer it.

Every design decision should also be tested against:

Would Ty personally use this every day?

The product has already touched notes, memory, travel, gifts, CRM, SBIRs, skills, resume updates, expenses, calendar, relationships, and research. That breadth is useful for future vision, but dangerous for V1.

The actual breakthrough is trusted external memory. Everything else depends on it:

- Travel recommendations need memory.
- Gift reminders need memory.
- Skills graph needs memory.
- SBIR tracking needs memory.
- Resume updates need memory.
- Relationship intelligence needs memory.

If the memory layer works for Ty, the rest becomes possible. If the memory layer does not work for Ty, none of the other features matter.

## Target User Assumptions

The first users are likely to be high-context individuals:

- Founders
- Consultants
- Operators
- Executives
- Creators

They likely share these traits:

- They have lots of ideas.
- They hate organizing.
- They use multiple devices.
- They want AI assistance.
- They want proactive suggestions.
- They carry context across people, companies, projects, documents, tasks, opportunities, and trips.

## V1 Assumptions

V1 should prove that users value:

- Capture.
- AI classification.
- Memory objects.
- Relationships.
- Search.
- One suggested action.

V1 should not prove whether Chief of Staff can become:

- Travel booking.
- Full CRM.
- Expense platform.
- Password manager.
- Enterprise collaboration.
- Native iPhone app.
- Multi-agent orchestration.

## Failure Questions

- What if users do not want proactive suggestions?
- What if users capture information once and never build a daily habit?
- What if AI organization feels less trustworthy than manual folders?
- What if users do not understand why the system remembered something?
- What if privacy concerns prevent users from adding useful context?
- What if relationship graphs are intellectually interesting but not practically useful?
- What if the product is too broad to explain in one sentence?
- What if the smallest useful version is smaller than the team wants to build?

## Next Strategic Questions

- Challenge this architecture and identify the three biggest reasons this startup fails.
- If the MVP had to be cut in half, what should be removed?
- What is the fastest path to getting 10 users who would use this every day?
