# Founder Context

Chief of Staff exists because the founder wants to spend less time organizing information and more time acting on it.

## Why This Exists

This product exists because Ty has repeatedly experienced the same problem across work, family, ideas, travel, learning, and future businesses.

This is not an attempt to build a better note-taking application.

This is an attempt to reduce cognitive load.

## Current Reality

Ty has many active and emerging contexts at once:

- ASEC
- Palms AI
- Family
- Travel
- Future businesses
- YouTube and content ideas
- Certifications
- Networking
- SBIR opportunities
- Articles, videos, podcasts, and research
- Software subscriptions
- Skills and career history

Ty constantly generates ideas. The problem is not idea generation.

The problem is:

1. Idea
2. Forgotten

## Core Frustration

The biggest frustration is not only finding information.

The biggest frustration is:

- Forgetting information.
- Losing ideas.
- Failing to follow up.
- Re-researching things.
- Not connecting related ideas.
- Not acting on opportunities already identified.
- Carrying too many open loops mentally.

## What Ty Does Not Enjoy

- Folder structures
- Tagging
- Knowledge management systems
- Manual organization
- Maintaining databases
- Choosing the perfect app for each kind of information
- Acting as his own librarian

## What Ty Wants

- One place to dump information.
- Automatic organization.
- Automatic relationship discovery.
- Automatic reminders.
- Automatic research assistance.
- Automatic context building.
- Cross-device access.
- A system that becomes more useful over time.

## Product Implication

Chief of Staff should function as trusted external memory and a user-controlled chief of staff.

The product should help Ty:

- Capture without friction.
- Remember what matters.
- Revisit ideas at the right time.
- See connections between people, projects, opportunities, and preferences.
- Reduce mental load.
- Spend more time with family and on meaningful work.
- Spend less time managing systems.

If a feature does not reduce cognitive load, question whether it should exist.

If a feature requires Ty to maintain a complex system manually, it is probably wrong for V1.

## Design Filter

Every design decision should answer:

Would this help Ty personally use the system every day?

Every feature should answer:

Does this reduce cognitive load or create more system-maintenance burden?
