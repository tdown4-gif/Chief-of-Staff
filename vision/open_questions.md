# Open Questions

## Product

- What is the smallest useful version of Chief of Staff?
- Which core objects matter most for V1?
- Should the universal inbox support only manual entry first, or include file upload from day one?
- How proactive should the system be before it becomes noisy?
- What should require explicit approval?
- What is the right balance between automation and user review?

## Memory

- What qualifies as a durable memory?
- When should the system create a memory versus keep something as a raw captured item?
- How should conflicting memories be handled?
- How should stale memories decay or become less prominent?
- How much source evidence does each memory need?
- What should the user be able to delete permanently?

## Trust And Privacy

- What data should never be used for proactive suggestions?
- What privacy controls are required before users trust the product?
- How should the system explain why it remembered something?
- How should sensitive objects such as subscriptions, people, and opportunities be protected?

## Go-To-Market

- Who is the first user: founders, operators, consultants, creators, researchers, or job seekers?
- Is this a personal productivity product, a founder operating system, or a memory infrastructure product?
- What is the wedge use case that makes someone return daily?
- What product promise is easiest to demonstrate in a short demo?

## V1 Boundaries

- How do we avoid drifting into a full CRM?
- How do we avoid building a calendar app?
- How do we support trips without becoming a travel booking product?
- How do we support documents without becoming a document manager?
- How do we show value without complex dashboards?
